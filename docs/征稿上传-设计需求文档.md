# 摄影征稿上传系统 · 设计需求文档（PRD）

> 版本：v0.1（草案）　日期：2026-06-15　负责人：amos
> 关联系统：Photo-Aiid-system（Tauri 桌面壳 + FastAPI + SQLite）

---

## 1. 背景与目标

### 1.1 背景
Photo-Aiid-system 目前是**纯本地（local-first）**应用：扫描本地文件夹 → AI 分析 → 批量重命名。
在图书编辑场景中，需要向**上百位摄影师**征集照片，由摄影师自行上传到云端，编辑端再统一拉取、进入既有的 AI 分析 / 重命名流程。

### 1.2 目标
- 摄影师**无需安装任何软件**，通过浏览器网页即可投稿。
- 支持**大文件**（RAW / 高清 JPEG，单张几十 MB，单人投稿可达 GB 级）。
- 单本书总量预计约 **100 GB**，上百位摄影师。
- 投稿原图**长期归档**，可随时取用。
- 采集**版权授权**与摄影师身份信息，满足图书出版举证要求。

### 1.3 非目标（本期不做）
- 摄影师账号体系 / 注册登录（用「投稿码」替代）。
- 在线选片、修图、协作评审。
- 付费 / 稿酬结算。

## 3. 存储架构

```
征稿中：摄影师 ──分片直传──> 阿里云 OSS（photo-intake，短期接收，国内快）
截稿后：OSS ──rclone 搬运──> Cloudflare R2（photo-archive，长期归档）
       └─ rclone check 核对一致后，清空 OSS 该书前缀
日常取用：编辑端 ──下载──> R2（多次取用免出流量）
```

### 3.1 选型结论
- **接收端：阿里云 OSS**。摄影师多在国内，上传速度快、并发稳。
- **归档端：Cloudflare R2**。长期存储便宜、**出流量免费**，适合反复下载。
- 关键事实：OSS → R2 复制时按 OSS 出流量计费一次（约 ¥50–80/100GB），无法规避；R2 的免出流量体现在**后续的反复下载**与**灾备冗余**。

### 3.2 桶与目录结构
```
OSS 桶: photo-intake          R2 桶: photo-archive
key 结构（两端一致，原样平移）：
  {书目代号}/{投稿码-摄影师名}/{主题}/{原始文件名}

例：
  2026-sanxia/
    ├─ A01-张伟/                          （按内容标注 content_label 归档）
    │    ├─ 瞿塘峡晨雾/DSC_0012.NEF
    │    ├─ 白帝城/IMG_3344.JPG
    │    └─ 夔门日出/SUNRISE_01.ARW      ← 数周后回访追加
    ├─ A02-李娜/古镇/IMG_8890.JPG
    └─ _meta/
         ├─ submissions.csv     （投稿与授权记录，自 SQLite 导出）
         └─ manifest.json       （文件清单 + 大小 + 校验值，搬运核对用）
```
- 顶层按**书目代号**聚合：搬运 / 归档 / 删除均为整前缀操作。
- 第二层按**投稿码-摄影师**：上百人天然分文件夹，便于催稿，投稿码在前保证唯一。
- 第三层按**内容标注**：照片按摄影师提交时填写的 `content_label` 归档。一位摄影师常提交多个不同内容的资料，且会**数周后找到照片再次回访追加**；按标注归档让同一摄影师的多批次、多内容各自成组、互不覆盖。
- `_meta/`：原图与授权记录同存一处，作为版权举证依据，不得分离。

> 交互与数据模型：摄影师**先把照片全部拉入**，再框选同类**批量标注内容**，所有照片标注完才能提交（强制）。一个投稿码（摄影师）→ 多次回访（每次 = 一条 `submissions`）→ 每张文件带 `content_label`（NOT NULL）。回访时上传页展示历史已交批次，并允许新增。

### 3.3 生命周期
- OSS：设置「超 N 天转低频/归档存储」兜底，防止遗忘搬运导致费用失控。
- 截稿后 `rclone copy` 搬至 R2 → `rclone check --checksum` 核对 → 通过后再清空 OSS 前缀（核对未通过严禁删除）。

## 4. 功能需求

### 4.1 摄影师上传页（公网）
| 编号 | 功能 | 说明 | 优先级 |
|---|---|---|---|
| U1 | 投稿码校验 | 进入页面输入/链接携带投稿码，校验有效性，绑定书目与摄影师 | P0 |
| U2 | 信息表单 | 姓名、联系方式（手机/邮箱）、拍摄说明（选填） | P0 |
| U3 | 版权授权 | 必须勾选授权声明后方可上传；记录授权时间 | P0 |
| U9 | 回访历史 | 校验投稿码后展示该摄影师**历史已交批次**（标注/张数/状态/日期），支持随时再次进入追加 | P0 |
| U10 | 先拉入后标注 | 照片先全部拖入形成**缩略图网格**；提交前**强制**为每张标注内容 | P0 |
| U11 | 多选批量标注 | 可框选多张同类照片，输入一次内容标注**批量应用**；未标注张数实时提示 | P0 |
| U12 | 强制校验 | 仍有未标注照片时提交按钮禁用，提示「还有 N 张未标注」 | P0 |
| U4 | 文件选择 | 点击选择 / 拖拽，多选；显示每个文件名、大小、缩略图 | P0 |
| U5 | 分片直传 OSS | 使用 OSS 前端 SDK 分片上传、断点续传、并发控制 | P0 |
| U6 | 进度反馈 | 单文件进度条 + 总进度；成功/失败状态；失败可重试 | P0 |
| U7 | 类型/大小校验 | 前端 + 服务端二次校验：白名单格式、单文件上限、总量上限 | P0 |
| U8 | 完成回执 | 上传完成展示提交编号与已收文件数；可继续追加上传 | P1 |

### 4.2 后端接口（扩展现有 FastAPI）
| 编号 | 接口 | 方法 | 说明 |
|---|---|---|---|
| A1 | `/submit` | POST | 校验投稿码 → 写摄影师/投稿元数据 → 返回 OSS 直传凭证（STS 临时授权或预签名） |
| A2 | `/submit/complete` | POST | 上传完成回调，记录文件 key / 大小 / 校验值 / 状态 |
| A3 | `/admin/submissions` | GET | 征稿看板数据：按书目列出谁交了/未交、各文件数与总量 |
| A4 | `/admin/codes` | POST/GET | 生成与查询投稿码（绑定书目与摄影师） |

> 安全：A1 不下发长期 AK，使用 **STS 临时凭证**或**服务端预签名 URL**，限定可写前缀 `{书目代号}/{投稿码-摄影师}/` 与有效期。

### 4.3 编辑端（桌面 App）
| 编号 | 功能 | 说明 | 优先级 |
|---|---|---|---|
| E1 | 征稿看板 | 复用 A3，查看进度、催稿名单 | P1 |
| E2 | 拉取本期投稿 | 从 R2/OSS 下载到本地指定文件夹 | P1 |
| E3 | 接入既有流程 | 下载完成后自动触发现有扫描/分析/重命名 | P2 |

## 5. 数据模型（SQLite 新增表）

```sql
-- 摄影师
CREATE TABLE photographers (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  book_code   TEXT NOT NULL,       -- 书目代号，如 2026-sanxia
  invite_code TEXT NOT NULL UNIQUE,-- 投稿码，如 A01
  name        TEXT,
  contact     TEXT,                -- 手机/邮箱
  created_at  REAL NOT NULL
);

-- 投稿（一次回访 = 一条会话；同一摄影师可多次回访）
CREATE TABLE submissions (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  photographer_id INTEGER NOT NULL,
  note          TEXT,              -- 整批备注（选填）
  license_agreed INTEGER NOT NULL DEFAULT 0,  -- 授权勾选
  license_at    REAL,             -- 授权时间
  status        TEXT NOT NULL DEFAULT 'open', -- open|completed
  created_at    REAL NOT NULL
);

-- 上传的单个文件（内容标注强制，提交前必填，可批量标注得到）
CREATE TABLE submission_files (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  submission_id INTEGER NOT NULL,
  content_label TEXT NOT NULL,     -- 照片内容标注，如「夔门日出」；NOT NULL = 强制
  object_key    TEXT NOT NULL,     -- OSS/R2 中的 key（按 content_label 归档）
  file_name     TEXT NOT NULL,
  file_size     INTEGER NOT NULL DEFAULT 0,
  checksum      TEXT,              -- 校验值
  status        TEXT NOT NULL DEFAULT 'uploading', -- uploading|done|error
  created_at    REAL NOT NULL
);
```

> 本人确认：所提交照片为本人原创并拥有完整著作权，不侵犯任何第三方权利；
> 同意授权 [出版方名称] 在 [图书名称] 及其相关宣传中**无偿/约定**使用本人提交的照片，
> 包括复制、发行、信息网络传播等必要权利；署名方式为：[本名 / 笔名 / 不署名]。
> 授权人：[摄影师姓名]　投稿码：[XX]　授权时间：[系统记录]

（具体权利范围、是否独家、稿酬等由编辑与摄影师另行约定，本系统仅做勾选与时间存证。）

## 7. 安全与合规
- 投稿码为唯一准入凭证，防止公网被乱传垃圾文件。
- OSS 写入使用 STS 临时凭证 / 预签名 URL，限定前缀与有效期，不下发长期密钥。
- 服务端二次校验文件类型与大小，拒绝可执行等危险类型。
- 全链路 HTTPS。
- 授权记录与原图同存归档，可追溯。

## 9. 里程碑（网页先上线）
1. **M1 选型与设计确认**（本文档）。
2. **M2 网页上传页 + 后端凭证接口最小可用版**（投稿码、回访历史、先拉入后标注、批量标注、强制校验、分片直传、看板）。**← 优先上线**
3. **M3 OSS→R2 搬运脚本与归档流程**（rclone 配置 + 核对 + 清理）。
4. **M4 桌面端拉取投稿、接入既有 AI 流程**（编辑端）。
5. **M5 桌面 App 投稿模式**（见第 11 章，后补，给核心摄影师）。

> 顺序原则：网页覆盖全部摄影师、零门槛，先稳上线；App 投稿模式作为进阶入口后补，不阻塞征稿启动。

## 11. 投稿客户端：网页 vs 桌面 App 投稿模式

摄影师有两条投稿路径，**共用同一套后端**（同一投稿码、`/submit` 凭证接口、OSS 目录结构、`submission_files.content_label`）。网页与 App 只是两个上传客户端，落库与归档逻辑完全一致。

### 11.1 两条路径定位
| | 网页上传页 | 桌面 App 投稿模式 |
|---|---|---|
| 摄影师准备 | 打开链接即用，零安装 | 下载安装投稿版 App（Win/Mac） |
| 内容标注 | 手动 + 多选批量标注 | **本地 CLIP 自动建议**，摄影师确认/修改 |
| 大批 RAW | 浏览器分片直传 | 本地先组织/筛选，再上传，体验更顺 |
| 适合人群 | 全部摄影师、临时投稿 | 投稿量大、愿装软件的核心摄影师 |
| 上线顺序 | **先上线（M2）** | 后补（M5） |

### 11.2 共用后端（关键）
网页和 App 投稿走同一接口与同一数据模型，编辑端无需区分来源：

```
网页客户端 ─┐
            ├─> /submit（投稿码 + STS 凭证）─> OSS {书}/{投稿码}/{标注}/ ─> submission_files
App 投稿模式 ┘
```

### 11.3 App 投稿模式功能裁剪
投稿版是现有 App 的**瘦身 + 锁权限**形态，**不等于**把完整编辑版发给外部：
- 保留：拉入照片 → 画廊缩略图 → 内容标注 → 凭投稿码上传。
- 隐藏：你的内部设置、引擎选择、重命名/导出、API key 配置等编辑功能。
- 入口：启动进入「投稿模式」，输入投稿码后只见投稿界面。

### 11.4 标注与密钥安全约束（硬性）
- **绝不在分发给摄影师的安装包内置任何云端 API key**（gemini/claude/zhipu），会被提取盗用、费用由你承担。
- 投稿模式 AI 标注**只用本地引擎**：`CLIP`（离线、免费、不泄密）给出建议标注，摄影师确认或修改；云端引擎在投稿版中禁用。
- 取舍：CLIP 自动建议需把本地模型打进投稿版（包体变大、首次需下载模型）。推荐采用——这是 App 投稿相对网页的核心价值。

### 11.5 维护提示
上百台摄影师机器、多版本、安装失败都会回流到你。因此：网页为主、覆盖全部人；App 投稿模式仅面向少数核心摄影师，按需发放，控制支持面。
