# 征稿工作流程（运营手册）

> 一本书从「开码」到「归档」的完整操作流程。配置细节见 [OSS接入配置.md](OSS接入配置.md)，
> 上线部署见 [上线操作手册.md](上线操作手册.md)。

---

## 0. 系统组成与数据流

```
摄影师浏览器 ──①投稿码校验──> 后端(FastAPI)
            <──②限定前缀临时凭证(STS, 1h)──
            ──③ali-oss 分片直传──> 阿里云 OSS(photo-intake)   ← 大文件不过后端
            ──④登记元数据────> 后端 → SQLite(submission_files)
后台看板 ──读──> SQLite                （监控各摄影师交稿进度）
截稿后 ──rclone copy+check──> Cloudflare R2(photo-archive) ← 长期归档
```

- **后端只做两件事**：发临时凭证、记元数据。原图直传 OSS，不过服务器，1 核 1G 可扛上百人。
- **安全隔离**：临时凭证用 RAM Policy 限定到 `{书}/{投稿码-姓名}/*`，摄影师只能写自己的目录，越权写入被 OSS 拒（403）。

---

## 1. 阶段一 · 准备（每本书一次）

1. 确认云端资源已就绪（见 [OSS接入配置.md](OSS接入配置.md)）：OSS 桶 + CORS、RAM 角色、发凭证子账号、R2 桶 + rclone。
2. 配置后端环境变量。本机走根目录 `.env`（已 gitignore，`backend/main.py` 启动时自动加载）：
   ```
   OSS_REGION / OSS_BUCKET / OSS_ACCESS_KEY_ID / OSS_ACCESS_KEY_SECRET / OSS_STS_ROLE_ARN
   INTAKE_ADMIN_TOKEN          # 后台登录口令，务必设置
   INTAKE_BOOK / INTAKE_BOOK_TITLE   # 书目代号与标题（默认 2026-sanxia / 三峡画册）
   ```
3. 启动：`python start.py`（前端 5173 + 后端 8000）。
4. 验证：`curl -s http://127.0.0.1:8000/api/intake/oss-config` 应返回 `{"mode":"oss",...}`。
   返回 `local` 说明 5 个 `OSS_*` 没配齐，会回退本地直存模式。

---

## 2. 阶段二 · 发投稿码

后台地址 `http://<域名>/intake/admin/codes`（首次访问输入 `INTAKE_ADMIN_TOKEN`，登录态存签名 Cookie 7 天）。

- **单个**：填姓名 → 生成投稿码（形如 `A01`、`A02`…按书目递增）。
- **批量**：粘贴多行姓名 → 一次性生成。
- 把「投稿链接 + 投稿码」发给对应摄影师，例：`http://<域名>/intake?code=A01`。

---

## 3. 阶段三 · 摄影师投稿（对方操作）

1. 打开投稿链接 → 输入/确认投稿码 → 显示「欢迎回来，{姓名}」。
2. 把照片**全部拖进**上传区。
3. **框选同类照片 → 输入内容标注（如「夔门日出」）→ 批量标注**。标注完才能提交。
4. 勾选版权授权 → 提交。浏览器分片直传 OSS，完成后登记元数据。
5. 可多次追加提交；看板累计计数。

> 校验点：浏览器 Network 中上传请求应直接打到 `https://{bucket}.{region}.aliyuncs.com`，
> 不经过后端。OSS 桶内出现 `{书}/{投稿码-姓名}/{标注}/文件名`。

---

## 4. 阶段四 · 监控进度

后台看板 `http://<域名>/intake/admin`：

- 各摄影师交稿张数、最近提交时间、未交名单一目了然。
- 数据来自 SQLite，可随时核对：
  ```bash
  cd backend && python3 -c "import sqlite3;c=sqlite3.connect('photo_aiid.db');\
  print('文件总数:',c.execute('SELECT count(*) FROM submission_files').fetchone()[0])"
  ```

---

## 5. 阶段五 · 截稿归档到 R2

截稿后把整本搬到 R2 长期归档（先只 copy+check，OSS 原样保留，安全）：

```bash
bash scripts/archive-to-r2.sh 2026-sanxia
```

脚本依次执行：
1. **[0/3]** 导出投稿/授权记录 → 上传 `{书}/_meta/submissions.csv`（版权举证依据）。
2. **[1/3]** `rclone copy`（断点续传、4 并发、校验）。
3. **[2/3]** `rclone check` 核对两端一致；**不通过绝不进入清理**。

核对输出 `0 differences found` 即成功（`N hashes could not be checked` 是 R2 分片上传的正常现象，不影响一致性判定）。

---

## 6. 阶段六 · 确认无误后清空 OSS（省存储费）

```bash
bash scripts/archive-to-r2.sh 2026-sanxia --purge
```

会再跑一遍 copy+check，通过后**二次确认（输入 yes）**才删 OSS 该书前缀。归档长期保留在 R2。

> 增量场景：归档后若又有新投稿，重跑不带 `--purge` 的命令即可只同步增量；确认后再 `--purge`。

## 命令速查

| 操作 | 命令 |
|---|---|
| 启动前后端 | `python start.py` |
| 确认 OSS 模式 | `curl -s localhost:8000/api/intake/oss-config` |
| 单独起后端 | `cd backend && python -m uvicorn main:app --host 127.0.0.1 --port 8000` |
| 停后端 | `pkill -f "uvicorn main:app"` |
| 看 OSS 内容 | `rclone ls oss:photo-intake/2026-sanxia/` |
| 看 R2 内容 | `rclone ls r2:photo-archive/2026-sanxia/` |
| 归档（保留 OSS） | `bash scripts/archive-to-r2.sh 2026-sanxia` |
| 归档并清空 OSS | `bash scripts/archive-to-r2.sh 2026-sanxia --purge` |

## 故障排查

| 现象 | 原因 / 处理 |
|---|---|
| `oss-config` 返回 `local` | 5 个 `OSS_*` 没配齐；检查 `.env` 或 export，重启后端 |
| 浏览器报 CORS 错 | OSS 桶跨域设置没加上传页域名（来源、Methods、Headers） |
| `STS 获取临时凭证失败` | 角色 ARN 错，或子账号缺 `AliyunSTSAssumeRoleAccess` |
| 上传 403 | RAM 角色对桶无写权限，或写到了非本人前缀（属预期拦截） |
| `rclone` 列桶 403 AccessDenied | 凭证限定了桶范围，属正常；改用 `rclone ls <remote>:<桶>` 直接访问具体桶 |
| 归档 `rclone ls oss:` 403 | 用于归档的子账号缺 OSS 读权限（需 `AliyunOSSReadOnlyAccess`/`Full`） |
| 后台无需口令直接进 | 没设 `INTAKE_ADMIN_TOKEN`（仅适合本机调试，上线务必设） |
