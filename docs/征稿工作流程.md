# 征稿工作流程（运营手册）

> 一本书从「开码」到「归档」的完整操作流程。配置细节见 [OSS接入配置.md](OSS接入配置.md)，
> 上线部署见 [部署到阿里云.md](部署到阿里云.md)。

> **架构现状（2026-06 起）**：网页征稿后端已**常驻阿里云服务器**（`intake.sdexp.org`，
> 精简版 `intake_server`，只含征稿接口），不再靠本机临时跑。日常管理（建书、发码、看板）
> 在**桌面 App 的「投稿管理」**里完成——它远程连接这台服务器；原网页后台 `/intake/admin*`
> 仍可用作备用。摄影师投稿、OSS 直传、R2 归档的机制不变。

---

## 0. 系统组成与数据流

```
摄影师浏览器 ──①投稿码校验──> 云端后端(intake.sdexp.org)
            <──②限定前缀临时凭证(STS, 1h)──
            ──③ali-oss 分片直传──> 阿里云 OSS(photo-intake)   ← 大文件不过后端
            ──④登记元数据────> 云端后端 → SQLite(submission_files)
编辑(桌面App「投稿管理」) ──远程管理(口令头)──> 云端后端   （建书/发码/看板）
征稿期间(定时) ──rclone copy 持续镜像──> Cloudflare R2(photo-archive) ← 异地备份
编辑随时 ──rclone copy 下载──> 本地        （优先从 R2 下载，流量免费）
```

- **后端只做两件事**：发临时凭证、记元数据。原图直传 OSS，不过服务器，1 核 1G 可扛上百人。
- **去重**：同一张照片的 OSS 路径固定（`{书}/{投稿码-姓名}/{标注}/{文件名}`），重复提交在 OSS 是覆盖、在库里按 `object_key` 更新而非新增，张数不会被重复计数。
- **安全隔离**：临时凭证用 RAM Policy 限定到 `{书}/{投稿码-姓名}/*`，摄影师只能写自己的目录，越权写入被 OSS 拒（403）。
- **存储策略**：**OSS 是长期主存**（摄影师直传 + 编辑下载的源）；**R2 是持续同步的异地备份**。征稿期间分钟级同步，不必等截稿。默认**不清空 OSS**（双云各留一份）。

## 1. 阶段一 · 准备

**云端后端只需部署一次**（已完成，见 [部署到阿里云.md](部署到阿里云.md)）：阿里云轻量服务器跑
`intake_server`，域名 `intake.sdexp.org`，`.env` 里配好 5 个 `OSS_*` + `INTAKE_ADMIN_TOKEN`。

- 验证服务在线：浏览器开 `https://intake.sdexp.org/intake` 显示投稿页；
  `curl -s https://intake.sdexp.org/api/intake/oss-config` 返回 `{"mode":"oss",...}`。
- 打开**桌面 App →「投稿管理」**→ 填服务器地址 `https://intake.sdexp.org` + 管理口令 → 连接。
  之后建书、发码、看板都在 App 内完成。

> 本机调试：仍可 `python start.py` 起本机后端，App 里把地址填 `http://127.0.0.1:8000` 即可。

---

## 2. 阶段二 · 建书 + 发投稿码（桌面 App）

App「投稿管理」连上后：

1. 左栏「书籍项目 ＋ 新建」→ 输书名（每本书一个项目）。
2. 进「投稿码管理」：
   - **单个**：填姓名（+可选联系方式）→ 生成投稿码（`A01`、`A02`…按书递增）。
   - **批量**：粘贴多行姓名 → 一次性连号生成。
   - 每行「复制链接」得到专属投稿链接 `https://intake.sdexp.org/intake?code=A01`（已带投稿码），
     发给对应摄影师；或「导出全部投稿码」下载 CSV 花名册群发。
3. 书籍头部「投稿地址」可「复制网页入口」群发，或「复制 App 服务器地址」给用桌面 App 投稿的摄影师。

> 备用：原网页后台 `https://intake.sdexp.org/intake/admin/codes`（输 `INTAKE_ADMIN_TOKEN` 登录）功能相同。

---

## 3. 阶段三 · 摄影师投稿（对方操作）

1. 打开投稿链接 → 输入/确认投稿码 → 显示「欢迎回来，{姓名}」。
2. 把照片**全部拖进**上传区。
3. **框选同类照片 → 输入内容标注（如「夔门日出」）→ 批量标注**。标注完才能提交。
4. 勾选版权授权 → 提交。浏览器分片直传 OSS，完成后登记元数据。
5. 可多次追加提交；看板累计计数。

> 校验点：浏览器 Network 中上传请求应直接打到 `https://{bucket}.{region}.aliyuncs.com`，
> 不经过后端。OSS 桶内出现 `{书}/{投稿码-姓名}/{标注}/文件名`。

---

## 4. 阶段四 · 监控进度

桌面 App「投稿管理」→ 选中书籍 →「征稿看板」：

- 顶部统计：摄影师数 / 已交 / 未交 / 收稿张数 / 占用容量。
- 表格：每位摄影师状态（已交/未交）、张数、大小、标注分布。点「刷新」拉最新。
- 备用网页看板：`https://intake.sdexp.org/intake/admin`。

---

## 5. 阶段五 · 征稿期间持续同步到 R2（不等截稿）

征稿一开始就让 R2 做异地备份镜像，分钟级准实时，编辑也能随时从 R2 下载。

**手动单次同步**（只增不删，OSS 原样保留）：
```bash
bash scripts/sync-to-r2.sh 2026-sanxia          # 增量同步
bash scripts/sync-to-r2.sh 2026-sanxia --check   # 同步后顺带核对一致性
```

**定时自动同步**（推荐，每 10 分钟，crontab）：
```bash
crontab -e
# 加一行（换成你的绝对路径）：
*/10 * * * * /bin/bash /绝对路径/scripts/sync-to-r2.sh 2026-sanxia >> /tmp/sync-r2.log 2>&1
```
脚本带 `mkdir` 原子锁，重叠自动跳过；增量同步靠 size+checksum 跳过已传文件，**只有新文件首次同步时产生一次 OSS 公网流量**，之后不重复烧流量。

> 费用：同步本身不会反复收流量费；**双云各存一份**，一本 50GB 约 OSS 6 元/月 + R2 4 元/月级别。详见本文末「成本与下载」。

---

## 6. 阶段六 · 编辑随时下载到本地

征稿过程中编辑选图、排版要把照片拉到本机。**优先从 R2 下载（下行流量免费）**：
```bash
rclone copy r2:photo-archive/2026-sanxia ~/三峡画册 --transfers 8 --progress
```
按摄影师、按分类下载，以及 Cyberduck 图形界面方式，见 [下载到本地教程.md](下载到本地教程.md)。

---

## 7. 阶段七 · 截稿后（可选）

- **保留双云**（推荐）：什么都不用做，OSS 主存 + R2 备份长期各留一份。
- **想省 OSS 存储费**：先用 `bash scripts/archive-to-r2.sh 2026-sanxia` 跑一遍 copy+check 确保 R2 完整，
  再 `bash scripts/archive-to-r2.sh 2026-sanxia --purge`（二次确认后）清空 OSS，只留 R2。
  - 注意：清空 OSS 后，编辑下载、版权举证都改从 R2 取；如仍需随时下载原图，建议保留 OSS。

> `archive-to-r2.sh` 还会导出投稿/授权记录到 `{书}/_meta/submissions.csv`（版权举证依据），随原图一起存档。

## 命令速查

| 操作 | 命令 |
|---|---|
| 验证云端在线 | `curl -s https://intake.sdexp.org/api/intake/oss-config` |
| 重启云端服务（改 .env 后） | `ssh root@47.236.198.127 'systemctl restart photo-intake'` |
| 看云端日志 | `ssh root@47.236.198.127 'journalctl -u photo-intake -f'` |
| 改了后端代码，推送到云端 | `rsync -avz backend/intake*.py root@47.236.198.127:/opt/photo-aiid/backend/` 后重启 |
| 本机调试起前后端 | `python start.py`（App 里填 `http://127.0.0.1:8000`） |
| 看 OSS 内容 | `rclone ls oss:photo-intake/2026-sanxia/` |
| 看 R2 内容 | `rclone ls r2:photo-archive/2026-sanxia/` |
| 看某本书占用 | `rclone size r2:photo-archive/2026-sanxia` |
| 持续同步到 R2 | `bash scripts/sync-to-r2.sh 2026-sanxia` |
| 下载到本地（从 R2，免费） | `rclone copy r2:photo-archive/2026-sanxia ~/三峡画册 --progress` |
| 截稿对账（保留 OSS） | `bash scripts/archive-to-r2.sh 2026-sanxia` |
| 截稿后清空 OSS（可选） | `bash scripts/archive-to-r2.sh 2026-sanxia --purge` |

## 故障排查

| 现象 | 原因 / 处理 |
|---|---|
| `oss-config` 返回 `local` | 5 个 `OSS_*` 没配齐；检查 `.env` 或 export，重启后端 |
| 浏览器报 CORS 错 | OSS 桶跨域设置没加上传页域名（来源、Methods、Headers） |
| `STS 获取临时凭证失败` | 角色 ARN 错，或子账号缺 `AliyunSTSAssumeRoleAccess` |
| 上传 403 | RAM 角色对桶无写权限，或写到了非本人前缀（属预期拦截） |
| `rclone` 列桶 403 AccessDenied | 凭证限定了桶范围，属正常；改用 `rclone ls <remote>:<桶>` 直接访问具体桶 |
| 归档 `rclone ls oss:` 403 | 用于归档的子账号缺 OSS 读权限（需 `AliyunOSSReadOnlyAccess`/`Full`） |
| 后台无需口令直接进 | 没设 `INTAKE_ADMIN_TOKEN`（仅适合本机调试，上线务必设） |
